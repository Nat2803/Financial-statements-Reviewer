# PRD: Проверка финансовой отчётности (Financial Statements Reviewer)

**Версия:** 1.0
**Статус:** реализовано (MVP), эксплуатируется
**Владелец:** —

---

## 1. Проблема

Сотрудники, готовящие финансовую отчётность, вручную ищут арифметические ошибки, несостыковки между формами отчётности и примечаниями, а также отклонения от МСФО (IFRS/IAS). Это трудоёмко, легко пропустить ошибку при большом объёме отчётности (200+ страниц), а профессиональный аудит доступен не на каждом этапе подготовки.

## 2. Цель

Дать сотрудникам инструмент для самостоятельной предварительной проверки отчётности перед сдачей — чтобы на выходе отчётность была без ошибок: без арифметических расхождений, без противоречий между формами и раскрытиями, без явных отклонений от МСФО.

**Критерий успеха:** сотрудник загружает пакет отчётности и за одну проверку получает структурированный список замечаний, устраняет их до финальной сдачи — без привлечения аудитора на этом этапе.

## 3. Пользователи

- Основная аудитория: сотрудники, которые готовят финансовую отчётность (бухгалтеры, специалисты по МСФО-отчётности).
- Не являются целевой аудиторией: внешние аудиторы, инвесторы, регуляторы (инструмент не заменяет их заключение).

## 4. Объём проверки

Проверка организована по разделам, результаты группируются по ним же:

1. **Баланс** — постатейно по каждой статье активов, обязательств и капитала:
   - арифметика и промежуточные/итоговые суммы (footing);
   - равенство Активы = Обязательства + Капитал;
   - корректность классификации текущих/долгосрочных статей (IAS 1);
   - логичность динамики статьи период к периоду.
2. **Отчёт о прибылях и убытках (ОПУ / SPL)** — постатейно:
   - арифметика и промежуточные итоги;
   - корректность классификации расходов (по функции / по характеру, IAS 1);
   - необычные или немотивированные отклонения.
3. **Учётная политика** — соответствие фактически применяемых методов (признание выручки, амортизация, оценка запасов, резервы и т.д.) заявленной политике, последовательность применения.
4. **Кросс-чеки раскрытий** — сверка цифр между всеми примечаниями и лицевыми формами отчётности:
   - совпадение сумм из примечаний со строками баланса и ОПУ;
   - совпадение одних и тех же показателей между разными примечаниями;
   - увязка чистой прибыли с изменением нераспределённой прибыли в капитале;
   - увязка остатка денежных средств (ДДС) с балансом.
5. **Общие замечания** — прочие логические противоречия и аномалии, не относящиеся к разделам выше.

Каждое замечание получает уровень критичности: **critical** (ошибка/расхождение) / **warning** (потенциальное несоответствие) / **info** (наблюдение, включая подтверждение того, что раздел проверен и сходится).

## 5. Функциональные требования

### 5.1 Загрузка данных
- Форматы: `.xlsx`, `.xls`, `.csv`, `.docx`, `.pdf`.
- Загрузка через drag-and-drop или выбор файла.
- Альтернатива — ручная вставка текста (например, скопированного из источника, который не поддерживается напрямую).
- Для `.xlsx`/`.xls`: поддержка нескольких листов, переключение между ними, предпросмотр таблицы.
- Для `.docx`: извлечение текста из документа.
- Для `.pdf`:
  - постраничное извлечение текстового слоя, до 200 страниц за один прогон;
  - автоматическое определение страниц без текстового слоя (сканы/фото форм) и распознавание их через OCR (рус/англ), до 30 страниц по умолчанию;
  - индикация прогресса обработки постранично и по OCR отдельно;
  - предупреждение, если часть страниц не обработана из-за лимита.
- Предпросмотр загруженных данных перед запуском проверки.

### 5.2 Проверка
- Запуск проверки по кнопке, с индикацией этапов («сверяем итоги баланса», «проверяем увязки между формами» и т.д.).
- Анализ выполняется через вызов LLM (Claude) с промтом, задающим структуру разделов и формат ответа.
- Ответ модели — строго структурированный JSON: сводные счётчики по критичности, общее заключение, список замечаний (раздел, статья, описание, ссылка на стандарт при наличии).
- Не более ~14 замечаний за прогон, отсортированных по критичности внутри каждого раздела.
- Модель не должна придумывать цифры — если данных для проверки раздела недостаточно, раздел пропускается.

### 5.3 Результаты
- Сводная строка счётчиков: критично / внимание / наблюдение.
- Общее заключение (1–2 предложения).
- Список замечаний, сгруппированный по разделам (см. п.4), с указанием статьи отчётности и ссылки на стандарт, если применимо.

### 5.4 Экспорт
- **PDF** — печать результатов в деловом оформлении через браузер (без элементов интерфейса загрузки).
- **Word (.doc)** — отдельный файл с той же структурой (заголовок, счётчики, заключение, замечания по разделам), с сохранением цветовой маркировки критичности, для приложения к пакету отчётности или дальнейшего редактирования.

## 6. Нефункциональные требования

- **Стиль:** деловой, без разговорных оборотов, конкретные формулировки с цифрами там, где есть расхождение.
- **Доступность:** инструмент — самостоятельный файл (веб-страница), запуск без установки, в браузере.
- **Приватность обработки файлов:** разбор `.xlsx`/`.docx`/`.pdf` (включая OCR) выполняется локально в браузере пользователя; во внешний API передаётся только извлечённый текстовый слой отчётности, необходимый для анализа.
- **Устойчивость к ошибкам:** при сбое чтения файла или вызова API — понятное сообщение об ошибке, без потери введённых данных.

## 7. Ограничения (известные)

- Инструмент не заменяет заключение аттестованного аудитора и не является профессиональной аудиторской/бухгалтерской консультацией — это прямо указано в интерфейсе.
- PDF без текстового слоя (сканы) распознаются через OCR; качество зависит от чёткости скана — возможны ошибки распознавания отдельных цифр, требуется выборочная ручная проверка.
- Обработка PDF ограничена 200 страницами, OCR — 30 страницами за прогон; превышение лимита фиксируется предупреждением.
- Объём передаваемого в модель текста ограничен (для очень больших пакетов отчётности анализируется не весь документ целиком, а его часть в пределах лимита).
- Максимум ~14 замечаний за один прогон проверки — при большом количестве ошибок может потребоваться несколько итераций (исправили — проверили снова).

## 8. Вне объёма (Out of scope)

- Полноценный аудит с формированием аудиторского заключения.
- Проверка соответствия иным стандартам, кроме IFRS/IAS (например, US GAAP, РСБУ).
- Многопользовательская совместная работа, история версий отчётности, хранение загруженных файлов.
- Автоматическое исправление найденных ошибок в исходном файле.

## 9. Технические заметки (как реализовано)

- Автономный HTML/JS-файл (без бэкенда), открывается в браузере.
- Разбор файлов на стороне клиента: SheetJS (xlsx/csv), mammoth.js (docx), pdf.js (текстовый слой PDF), Tesseract.js (OCR сканов).
- Анализ отчётности — вызов Anthropic API (Claude) с фиксированным системным промтом, ответ в формате JSON.
- Экспорт в PDF — через нативную печать браузера; экспорт в Word — генерация HTML-документа с MIME-типом `application/msword`.

## 10. Дальнейшее развитие (кандидаты, не приоритизировано)

- Повышение лимитов OCR/страниц PDF при необходимости.
- Поддержка дополнительных стандартов отчётности.
- Возможность донастройки чек-листа проверок под внутренние политики компании.
- Сохранение истории проверок одного и того же пакета отчётности (до/после исправлений).
